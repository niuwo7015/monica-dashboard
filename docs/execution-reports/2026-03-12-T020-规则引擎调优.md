## 问题背景

Dry-run结果显示 R3=0, R4=9,284，两项均不合理：
- R3（沉默激活）=0：应有大量客户触发
- R4（初始触达）=9,284：绝大多数是僵尸号/供应商/同事

## 诊断结果

### R3=0 根因：规则优先级覆盖Bug

`apply_rules()` 中规则按 R1→R2→R3 顺序检查，R1和R2都有 `return tasks` 提前返回：
- silence >= 7天 + last_sender=customer → R1命中（silence>=1天），提前返回，R3不可达
- silence >= 7天 + last_sender=sales → R2命中（silence>=3天），提前返回，R3不可达
- **R3永远不可能被执行到**

### R4=9,284 根因：僵尸号无法过滤

诊断数据：
- contacts总数: 13,393（非删除）
- contact_tag: 全部为"未分类"（无法区分客户类型）
- add_time: 全部为NULL（无法按加好友时间过滤）
- yunke_create_time: 全部为NULL
- 有非系统消息聊天记录的contacts: 3,455 / 13,393（仅26%）

### 附加问题：is_system_msg未过滤

`fetch_last_messages_for_sales()` 未排除系统消息（加好友通知等），可能导致silence_days计算偏小。

## 修复内容

### 1. R3修复：调整规则检查顺序

将R3（silence>=7天）移到R1/R2之前检查：
- silence >= 7天 → R3 reactivate（对话已"冷掉"，需激活）
- silence 1-6天, 客户末条 → R1 urgent_reply
- silence 3-6天, 销售末条 → R2 follow_up_silent

### 2. is_system_msg过滤

在 `fetch_last_messages_for_sales()` 查询中添加 `.eq('is_system_msg', False)`，排除系统消息对silence_days的干扰。

### 3. R4僵尸号过滤

由于 add_time/contact_tag/yunke_create_time 全部无数据，唯一可行方案：
- 只扫描有至少1条非系统消息聊天记录的客户
- contacts无聊天记录 → 跳过（不生成任何任务）
- R4暂停，等add_time数据可用后恢复新客触达功能
