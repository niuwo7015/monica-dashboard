# S-005: records接口私聊回补验证与执行

> 执行时间：2026-03-08 22:35–进行中
> 状态：**回补执行中**（后台运行）

---

## 1. 任务目标

验证云客 `/open/wechat/records` 接口能否用 `friendWechatId=好友微信号`（而非群ID）拉取私聊历史记录。如果可行，立即开始全量回补。

## 2. 背景

- S-004 发现 records 接口的 `friendWechatId` 参数暗示支持私聊
- 现有 allRecords 接口按1小时时间窗口扫描，效率低（大量空窗口）
- chat_messages 表现有约 4,323 条记录，覆盖不完整
- API 到期日 3月20日，剩余12天

## 3. 验证结果

### 3.1 探索脚本验证（yunke_explore_records_dm.py）

选取3个销售的3个好友进行测试，**全部成功返回私聊消息**：

| 好友 | 销售 | records返回 | 数据库已有 |
|------|------|------------|-----------|
| P25.3.1 懒懒 | 可欣(乐乐) | 30条 | 0条 |
| G25.3.1钢牙 | 可欣(乐乐) | 10条 | 0条 |
| 25.6.11 | 小杰(jay) | 30条 | 0条 |

**关键发现：**
1. records 接口 **确认支持私聊**，传好友微信号即可
2. 不带时间范围参数即可返回消息（默认返回全部可用历史）
3. 数据库中这些好友私聊消息为 **0条** — allRecords 完全没覆盖到
4. records 接口返回的 `talker` 字段正确区分销售/客户微信号
5. `hasNext` + `end` 分页机制与群聊完全一致

### 3.2 对比 allRecords

| 维度 | allRecords | records |
|------|-----------|---------|
| 拉取方式 | 按1小时时间窗口扫描 | 按好友维度直接拉取 |
| 效率 | 大量空窗口浪费调用 | 直接获取该好友全部消息 |
| 私聊覆盖 | 验证发现有遗漏 | 完整覆盖 |
| 参数 | createTimestamp | friendWechatId + wechatId |
