# Monica销售AI系统 · 修改后执行路径

> 基于20位专家圆桌会议结论，整合四层架构设计与落地策略
> 日期：2026年3月6日

---

## 核心原则（专家共识）

1. **先补数据，再选模型** — 输入数据的完整度决定输出质量，模型差异是次要的
2. **先养习惯，再加AI** — 销售对系统的第一印象不可逆，用简单准确的东西建立信任
3. **用"赚钱"驱动，不用"任务"驱动** — 所有推送和UI包装成机会，不是指令
4. **行为做反馈，不问销售** — 执行=正反馈，忽略=负反馈，零填写零输入
5. **存一切** — 所有API调用的原始input/output都存下来，数据是未来的资产
6. **放弃自研模型** — 调用量不支持自建部署的ROI，全部通过API实现

---

## 阶段总览

| 阶段 | 周期 | 核心目标 | AI参与度 |
|------|------|----------|----------|
| Phase 0 · 地基 | 第1-2周 | 安全修复 + 数据入库 + 基础设施 | 0% |
| Phase 1 · 养习惯 | 第3-4周 | 纯规则清单上线，销售每天打开系统 | 0% |
| Phase 2 · 偷跑验证 | 第5-8周 | 后台跑AI对比规则，数据决定是否上线AI | 后台静默 |
| Phase 3 · AI上线 | 第9-12周 | Opus先建信任，再A/B测试切DeepSeek | 前台可见 |
| Phase 4 · 深化 | 第13周+ | 素材推荐、深度分析按钮、企业微信评估 | 全面 |

## Phase 0 · 地基（第1-2周）

### P0-1：安全修复（第1天）

**问题：** Supabase service_role key硬编码在前端源码中，等于数据库对全网裸奔。

**修复方案：**
- 前端只使用anon key
- service_role key移到阿里云服务器环境变量
- 启用Supabase RLS（Row Level Security）策略
- 所有敏感操作通过服务端API中转

**执行：** Code直接改代码，1天完成。

---

### P0-2：云客聊天记录入库（第1-4天）

**目标：** 云客API → chat_messages表，全自动。

**表结构修改：**
- chat_messages新增 `file_url` text（媒体文件链接）
- chat_messages新增 `msg_svr_id` text（微信消息唯一ID）
- msg_svr_id 建唯一索引用于去重

**字段映射：**

| 云客API | chat_messages | 转换逻辑 |
|---------|---------------|----------|
| msgSvrId | msg_svr_id | 直接存，用于去重 |
| talker | wechat_id | 客户微信ID |
| wechatId | sales_id | 通过销售微信号映射 |
| mine=true | sender_type='sales' | 布尔转字符串 |
| mine=false | sender_type='customer' | 布尔转字符串 |
| text | content | 消息内容 |
| type | msg_type | 1文本,2图片,3语音,4视频,8GIF,9文件,10链接 |
| timestamp | sent_at | 毫秒时间戳转timestamp |
| file | file_url | 媒体文件URL |

**定时任务（部署到阿里云cron）：**
- 每小时拉增量聊天记录
- 历史全量回补脚本（自动跑，中断自动续，无需人盯）

---

### P0-3：订单数据导入（第3-4天）

**来自9号专家的强调：** 订单数据不只是成交记录，是理解"什么客户买什么产品"的关键维度。没有订单数据，AI缺一条腿。

**执行：** Monica提供历史订单Excel → 导入orders表 → 关联customer_id

---

### P0-4：服务器升级（第1天）

阿里云轻量服务器从2核1G升到2核4G。控制台直接升级，数据不丢，IP不变。

### P0-5：587个成交客户回溯分析（第1-2周并行）

**来自6号专家的建议：** 人工看30个典型案例，找出从加微信到成交的3-5种典型路径，作为后续AI诊断和规则引擎的基准模版。

**方法：**
- 从587个成交客户中抽30个（覆盖三个销售、不同产品线、不同客单价）
- 人工回看聊天记录，标注关键转折点
- 输出：3-5种"成交路径模版"（如：快速成交型、长周期培育型、比价后回头型）

**价值：** 先知道"正确答案长什么样"，才能评判AI输出对不对。

## Phase 1 · 养习惯（第3-4周）

### 核心策略：纯规则，零AI，100%准确率建立信任

**来自12号专家的核心建议：** 系统上线第一天推出的东西，决定了它在销售心目中的定位。第一版不要有AI诊断，用100%准确的简单规则赢得信任。

---

### P1-1：规则引擎（按装修阶段区分）

**基础规则：**
- 新客加微信未回复 → 必跟
- 客户发消息销售没回 → 必跟
- 已成交/已删除/供应商 → 排除
- 冻结客户 → 排除

**装修阶段差异化（来自14号专家）：**

| 装修阶段 | 沉默几天触发"该跟" | 跟进频率 |
|----------|-------------------|----------|
| 软装阶段（3个月内要买） | 3天 | 高频 |
| 硬装阶段（3-6个月） | 7天 | 中频 |
| 早期了解（6个月+） | 14天 | 低频 |
| 未知 | 5天（默认） | 中频 |

**装修进度数据来源：**
- Phase 1阶段：销售首次沟通后手动标记一次（14号方案）
- Phase 2阶段：DeepSeek从聊天记录自动提取装修关键词回填（14号最终建议）

---

### P1-2：首次沟通标记（最小阻力设计）

**来自18号的要求：** 只弹一次，2秒选完，以后不要再问。

**实现：** 销售首次跟新客户沟通后，客户卡片顶部出现一次性选择：

```
这个客户大概什么时候买家具？
[3个月内]  [半年内]  [只是看看]
```

选完消失，永不再问。数据写入customers表的`renovation_stage`字段。

---

### P1-3：SalesToday PWA化（来自19号设计方案）

**来自18号的硬性要求：** 必须手机能用，打开不用登录，3秒直达清单。

**改造：**
- 添加PWA manifest，可"添加到主屏幕"
- Token缓存，打开即用，不需要每次登录
- 首屏只显示一个大数字 + 客户列表

**交互设计（来自19号）：**
- 打开首屏：`今天有3个客户可能快成交了` + 排序后的客户头像列表
- 信息三层层级：
  - 第一眼：客户名 + 动作词（"该聊了"/"快成交"/"跟价格"）
  - 点进去：3句话客户近况（纯事实，零AI术语）
  - 再点：完整对话历史
- 左滑 = 跳过（系统记录为"忽略"）
- 右滑 = 去聊（系统记录为"执行"）
- 云客API检测到新消息 → 客户卡片自动变"已跟"

**90%的使用在第一层就结束。** 15个客户1分钟扫完。

---

### P1-4：飞书推送（赚钱包装）

**来自12号的关键建议：** 用"赚钱"驱动行为，不用"任务"驱动。

**推送文案对比：**

| ❌ 错误 | ✅ 正确 |
|---------|---------|
| 你今天有12个客户需要跟进 | 你有3个客户最近在活跃，可能快成交了 |
| 以下客户超过3天未跟进 | 张姐昨天又看了你的朋友圈，要不要聊聊？ |
| 请完成今日跟进任务 | 今天的赚钱机会AI帮你整理好了 |

**推送结构：**
- 标题突出最有希望的2-3个客户（机会感）
- 点击链接直接打开SalesToday PWA
- 销售看了热门客户，顺便看到其他人 → 覆盖率自然提升

---

### P1-5：晚间执行追踪 + Monica报告

**每天21:00自动跑：**
- 检查清单上的客户，销售今天有没有发消息（云客API数据）
- 生成执行率：`小杰跟了12/15，可欣跟了6/13，霄剑跟了10/11`
- 飞书推送给Monica

**Monica看到的不是"谁没完成任务"，而是"谁今天努力了"。** 正向表达。

---

### P1-6：验证里程碑（来自15号）

| 时间点 | 指标 | 目标 | 未达标行动 |
|--------|------|------|-----------|
| 第2周 | 覆盖率 | 39% → 55% | 检查推送到达率和打开率，改UI |
| 第4周 | 覆盖率 | → 65% | 检查清单质量，是否太多误报 |
| 第4周 | 执行率 | > 70% | 检查销售阻力来源，调整包装 |
| 第4周 | 销售日均打开次数 | > 1次 | 习惯未养成，先不进Phase 2 |

**如果第2周覆盖率没到55%，不是模型的问题，是产品形态的问题——回来改UI和推送方式。**

## Phase 2 · 偷跑验证（第5-8周）

### 核心策略：后台静默跑AI，用数据决定是否上线

---

### P2-1：后台偷跑DeepSeek诊断

**每天早上规则清单生成后，同时对同一批客户跑DeepSeek V3.2 thinking模式：**
- 输入：客户最近聊天记录 + 客户基本信息 + 装修阶段
- 输出：状态摘要 + 建议动作 + 置信度
- 结果存入新表 `ai_diagnostics_shadow`（销售看不到）

**同时存原始prompt和完整返回（来自16号建议）。**

---

### P2-2：DeepSeek自动提取装修进度

**来自14号最终建议：** 扫描聊天记录，自动提取装修进度关键词。

**关键词示例：**
- "刚拿钥匙""还没开始装" → 早期
- "硬装在做""水电改完了""瓦工在贴砖" → 硬装阶段
- "软装""选家具""设计师推荐" → 软装阶段
- "下个月搬""马上入住" → 紧迫

**自动回填customers表，补全销售未手动标记的客户。**

---

### P2-3：交叉验证客户状态（来自5号）

**从聊天记录行为推断客户真实阶段：**
- 客户声称"在装修"，但3周无主动消息、未问过任何产品细节 → 大概率早期了解，标记"行为与声明不符"
- 真正在装修的客户会问工期、问材质、发户型图 → 行为验证

**这个判断DeepSeek就能做，不需要Opus。** 纯行为模式匹配。

---

### P2-4：语气温度监测（来自6号）

**对所有客户对话跑一个"语气温度"指标：**
- 监测回复长度、回复速度、主动话题发起频率的变化趋势
- 从长文本+主动提问 → 突然变成"嗯""好的""收到" → 标红
- 温度突变的case自动标记，作为Phase 3 Opus分析的候选

**成本几乎为零，但能抓住规则抓不到的隐性信号。**

---

### P2-5：规则 vs AI 对比报告（来自15号）

**第6周出对比报告：**
- 规则推荐跟进的客户 vs DeepSeek推荐跟进的客户，重叠率多少？
- DeepSeek额外推荐的客户，事后看有没有成交/回应？
- DeepSeek遗漏的客户（规则推了但AI没推），有没有问题？

**决策标准：**

| 差异率 | 结论 | 行动 |
|--------|------|------|
| < 20% | 规则够用，AI价值不大 | 暂不上线AI，继续积累数据 |
| 20%-40% | AI有增量价值 | Phase 3上线AI |
| > 40% | AI能捕捉到规则抓不到的东西 | Phase 3加速上线 |

---

## Phase 3 · AI上线（第9-12周）

### 核心策略：先用Opus建信任，再切DeepSeek降成本

---

### P3-1：模型调用架构

**来自4号AI产品经理的最终方案 + 12号的"先建信任"策略：**

**第9-12周（建信任期）：全用Opus 4.6**

| 场景 | 模型 | 调用模式 | 单次成本 |
|------|------|----------|----------|
| 每日状态摘要 | Opus 4.6 标准模式 | 批量（Batch API 50%off） | ~$0.05 |
| 高价值客户深度分析 | Opus 4.6 extended thinking | 按需触发 | ~$0.30-0.50 |

每天40个客户全跑Opus，月成本约$100-150（约¥700-1100）。用这个钱买销售信任。

**第13周起：A/B测试切换**
- 同一批客户同时跑Opus和DeepSeek V3.2 thinking模式
- 对比诊断差异率
- 差异率<15%：安全切DeepSeek，常规诊断成本降到$5-10/月
- 差异率>30%：继续用Opus

---

### P3-2：路由逻辑（行为规则，非模型自评）

**来自16号NLP研究员的方案：** 用确定性规则捕捉不确定性信号，不依赖模型自评confidence。

**自动推送到Opus extended thinking的触发条件：**
- 客户沉默后主动回来（心理状态可能转变）
- 客户聊天频率突然下降（兴趣可能变化）
- 客户连续问3个以上价格相关问题（可能在比价）
- 客户对话中出现决策人词汇（"老公/老婆/爸妈"）→ 多人决策
- 语气温度突变（Phase 2监测发现的标红case）

**其余客户走标准模式即可。**

---

### P3-3：输出语言规范（来自18号的硬性要求）

**系统永远不要输出AI术语。**

| ❌ AI话 | ✅ 人话 |
|---------|---------|
| 客户处于决策犹豫期，建议用紧迫性话术推进 | 张姐3天前问了价格后没回复，她之前说下个月搬新家 |
| 诊断：价格异议阶段，confidence 0.72 | 张姐问了两次价格，可能在比其他家 |
| 建议probe_value类型跟进 | 可以聊聊她家的风格想法，不急着报价 |

**原则（来自16号）：** 系统永远不做猜测性判断（"老公应该回来了"），只呈现事实和提醒（"14天前提到老公出差，建议确认进展"）。把判断权交给销售。

---

### P3-4：隐式反馈机制（来自19号）

**不做反馈按钮，做执行动作：**
- 销售右滑"去聊" → 系统记录为"采纳建议"
- 销售左滑跳过 → 系统记录为"忽略建议"
- 销售执行后客户有回应 → 标记为"有效建议"
- 销售执行后客户无回应 → 标记为"无效建议"

**数据自动积累到 `follow_ups` 表，形成"建议→执行→结果"闭环。**

## Phase 4 · 深化（第13周+）

### P4-1：素材库上线

**前置条件：** 第二层状态摘要已跑通，系统知道客户在什么阶段。

**实现：**
- 素材存阿里云OSS，元数据存Supabase `materials`表
- AI诊断时顺带输出"该客户适合什么类型的素材"
- SalesToday客户卡片下方显示"推荐素材"缩略图
- 销售点击 → 大图预览 → 长按保存到手机相册 → 切微信发

---

### P4-2：销售"问AI"按钮

**来自原四层架构第四层的按需触发版本：**
- SalesToday客户详情页加"问AI"按钮
- 点击 → 拉该客户最新聊天记录 → Opus extended thinking深度分析
- 返回3句话：客户心理状态、当前策略对不对、建议方向
- **语言标准同P3-3，必须是人话**

---

### P4-3：企业微信迁移评估

**前置条件：** 云客服务续期后系统稳定运行，Phase 1-3验证完毕。

**评估内容：**
- 新客用企业微信加，老客保持个人微信
- 企业微信侧边栏 + 素材库一键发送
- 会话存档替代云客API
- 客户离职交接

**需要Monica拍板的决策，准备一页纸对比给她看。**

---

### P4-4：巨量引擎数据打通（来自5号和9号）

**目标：** 新客加微信时，销售手机5分钟内弹出画像——从哪来、看了什么产品、停留多久。

**技术链路：** 巨量引擎线索表单 → 手机号 → 云客/企微好友匹配 → 客户画像自动生成

**优先级排在Phase 4是因为需要巨量引擎API对接，且依赖客户手机号桥接。**
